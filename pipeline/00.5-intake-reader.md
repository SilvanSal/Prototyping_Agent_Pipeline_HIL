# Stage 00.5 — Intake Reader + Early Clarification

**Run by:** `Intake-Reader` subagent (fresh context, read-only + Write for the two output files)
**Reads:** `input/**`, `specs/constitution.md`
**Produces:** `specs/intake-brief.md`, `specs/intake-qa.md`

## Purpose

The first agent the user interacts with after constitution. It reads everything the user has placed in `input/`, synthesises a structured brief, and asks 5–10 targeted questions that surface non-obvious ambiguities — the kind that would send the architecture in materially different directions but are not stated anywhere in the input materials.

This stage eliminates generic research and clarification. Stage 01 (Domain-Researcher) operates as a targeted follow-up, not a blank-slate survey.

## What the Intake-Reader does

### Step 1 — Read all input materials

Read every file in `input/**`. For PDFs, use the pdf skill to extract full text. For images (wireframes, mockups), treat them as visual references and describe what you see. For `.yaml`/`.json` specs, read them in full.

If `input/` is empty or contains only `README.md`, note this and proceed: the synthesis sections will be sparse, but the Q&A gate is still valuable.

### Step 2 — Produce the synthesis

Write `specs/intake-brief.md` from `templates/intake-brief.md` with these 6 sections:

1. **Problem statement** — 1 paragraph in the user's own words (quote phrases directly from input materials where possible). Do not reframe or improve.
2. **Stated constraints** — explicit requirements from the materials: technology choices already committed, compliance requirements, deadlines, budget signals. Bullet list.
3. **Implied user types** — personas inferred from the materials. Mark each as `[inferred]` or `[stated]`. Do not conflate the two.
4. **Referenced prior art** — papers, products, APIs, standards explicitly cited in the input materials. List name + brief context.
5. **Identified conflicts** — places where two input documents contradict each other. Name both documents and the conflicting claims. If none found, write "none identified."
6. **Unstated assumptions** — things the materials seem to assume without stating (e.g., "assumes users have desktop browsers", "assumes English-only"). Be precise; do not speculate wildly.

### Step 3 — Generate questions

Write `specs/intake-qa.md` from `templates/intake-qa.md`. Generate 5–10 questions with these rules:

- Each question targets a gap that would **materially change the architecture, scope, or user model** if answered differently
- Do NOT ask about things the materials already answer clearly
- Do NOT ask about implementation details (that is Stage 06's job)
- Each question has **4–5 labeled options** (A/B/C/D or A/B/C/D/E)
  - One option is marked `(recommended)` with a 1-sentence rationale for why it is the common-sense default
  - Options should be meaningfully different, not paraphrases of each other
- Questions are ordered: most architecturally consequential first
- **Maximum one question per topic area** — if two questions both touch auth, keep only the more impactful one
- Stop at 10 questions; additional gaps go into the synthesis's "Unstated assumptions" section

### Step 4 — Wait for user answers

Present the synthesis and the questions. Wait. Do not proceed to Stage 01 until all questions have answers.

If the user selects an option and adds free-text qualifications (e.g., "B, but with the caveat that..."), record both: the option letter and the free-text. Both are binding.

If the user says "skip" or "doesn't matter" for a question, record that explicitly. Stage 04's Architect reads this and makes the default choice.

### Step 5 — Record answers

Update `specs/intake-qa.md` to include the user's answer for each question. Write `specs/intake-qa.md` as a single file that contains both the questions and the answers — it is the Q&A record, not just the questions.

## Output format

### `specs/intake-brief.md`

```markdown
# Intake Brief
_Intake pass: [YYYY-MM-DD]_

## Problem statement
[1 paragraph, user's own words]

## Stated constraints
- [constraint 1]
- [constraint 2]

## Implied user types
- [Persona name] — [description] — [inferred/stated]

## Referenced prior art
- [Name] — [context]

## Identified conflicts
- [Source A] vs [Source B]: [what conflicts]
- (or "none identified")

## Unstated assumptions
- [Assumption]
```

### `specs/intake-qa.md`

```markdown
# Intake Q&A
_Intake pass: [YYYY-MM-DD]_

## Q1: [Short title — the architectural decision this targets]

[1 sentence explaining why this decision matters]

A. [Option A]
B. [Option B] (recommended — [1-sentence rationale])
C. [Option C]
D. [Option D]

**Answer:** [user's selection, or "skipped"]
[free-text qualifications if any]

## Q2: ...
```

## Hard rules

- Do NOT propose an architecture or tech stack.
- Do NOT answer questions on the user's behalf.
- Every question in `intake-qa.md` must have an answer before Stage 01 starts.
- `intake-brief.md` may not claim something is "stated" unless you can cite a specific input document.
- If the user hasn't placed anything in `input/`, the synthesis sections will be sparse — that is correct; do not hallucinate content.

## Orchestrator dispatch prompt (copy verbatim)

> You are the Intake-Reader subagent. Fresh context. Read all files under `input/**` (use pdf skill for PDFs, treat images as visual descriptions). Also read `specs/constitution.md`.
>
> Your job:
> 1. Produce `specs/intake-brief.md` with a 6-section synthesis per `pipeline/00.5-intake-reader.md`.
> 2. Produce `specs/intake-qa.md` with 5–10 questions in A/B/C/D format per the rules in `pipeline/00.5-intake-reader.md`.
> 3. Present the synthesis and questions to the user. Wait for answers. Do not proceed to Stage 01 until all questions are answered.
> 4. Record answers in `specs/intake-qa.md`.
>
> You may NOT propose an architecture, pick a tech stack, or write code. Stop after writing both files with answers recorded.

## Stop condition

`specs/intake-brief.md` exists with all 6 sections (even if sparse). `specs/intake-qa.md` exists with 5–10 questions, each with a recorded user answer. Both files are committed.
