# Stage 04.5 — Phase Planning

**Run by:** `Phase-Planner` subagent (fresh context)
**Reads:** `specs/[feature]/requirements.md`, `specs/[feature]/design.md`, `specs/[feature]/eval-spec.md`, `specs/intake-qa.md`, `specs/constitution.md`
**Produces:** `specs/[feature]/phase-plan.md`

## Purpose

Phase planning sits between architecture (04) and slice planning (05). It answers: "what are the milestone releases that deliver user value incrementally?" Phases are user-visible milestones (MVP, v1.0, v1.1); slices are implementation chunks within a phase.

Without this stage, slice planning operates on the entire feature at once, which can produce 15+ slices with no milestone structure. Phase planning constrains scope per run.

## What the Phase-Planner produces

Use `templates/phase-plan.md`. The file contains:

1. **Phase list** — 2–5 phases, each with: ID, name, user outcome, requirements satisfied, acceptance criteria, estimated slice count, dependencies on prior phases.
2. **"Not in scope" section** — requirements explicitly deferred to future work.
3. **Phase ordering rationale** — why P1 is the MVP, what each phase unblocks.
4. **Selected phase** — filled in after user approval.

## Rules for this stage

- Every phase must be user-visible and independently shippable. No "set up infrastructure" phases.
- Phase 1 is always the MVP: smallest thing that delivers core user value.
- Highest user value with fewest dependencies first.
- No implicit deferral — if a requirement isn't in a phase, it must be in the "Not in scope" section.
- If the feature is simple enough for a single phase, write a single-phase plan. Do not invent artificial phases to hit a minimum count.

## User gate

After the Phase-Planner produces `phase-plan.md`, the orchestrator presents it to the user. The user:
- **Approves** the full plan, OR
- **Rejects** with changes (Phase-Planner revises), OR
- **Approves Phase 1 only**, marks the rest "TBD".

Then the user selects which phase to build. The orchestrator records the selection in `phase-plan.md` and passes it to Stage 05.

## Orchestrator dispatch prompt (copy verbatim)

> You are the Phase-Planner subagent. Fresh context. Read: `specs/[feature]/requirements.md`, `specs/[feature]/design.md`, `specs/[feature]/eval-spec.md`, `specs/intake-qa.md`, `specs/constitution.md`.
>
> Your job: produce `specs/[feature]/phase-plan.md` using the skeleton in `templates/phase-plan.md`. Decompose requirements into 2–5 user-visible milestone phases. Phase 1 must be the MVP. Every requirement must appear in a phase or in the "Not in scope" section.
>
> Do NOT plan slices, sub-tasks, or code. Do NOT read domain research or code-style files.
>
> When done, output the file path and the phase list. Stop and wait for user approval.

## Stop condition

`specs/[feature]/phase-plan.md` exists with phases mapped to requirements. The user has approved the plan and selected a phase. The "Selected phase" section is filled in.

## After this stage

Stage 05 (Slice Planning) reads `phase-plan.md` and restricts its output to the selected phase's requirements only.
